# Implementation Plan

Рабочий план по этапам 1–6. Этап 7 с AI-агентами не входит в текущую разработку.

## Главный принцип

До AI-ключей дневник должен стать надёжным локальным инструментом:

- понятно, где лежат данные;
- легко сделать резервную копию;
- сложно случайно потерять записи;
- приватность не должна быть только нарисованной в интерфейсе;
- все важные функции должны работать без внешних сервисов.

## Порядок внедрения

### 1. Стабилизировать проект

Что уже сделано:

- базовая документация;
- roadmap;
- changelog;
- модель данных;
- архитектурные решения;
- npm-скрипты для dev/check.

Следующее:

- завести git-репозиторий;
- регулярно фиксировать изменения;
- держать `CHANGELOG.md` в актуальном состоянии.

### 2. Отделить данные от UI

Цель: подготовиться к IndexedDB без переписывания экранов.

Порядок:

1. Вынести чистые функции дат, тегов и статистики.
2. Вынести normalize/import/export.
3. Добавить storage repository.
4. Перевести UI на repository API.
5. Добавить IndexedDB adapter.
6. Сделать миграцию из `localStorage`.

### 3. Локальная автоматизация

Внедрять до приватности, но без усложнения схемы данных.

Приоритет:

1. Статус автосохранения: `сохраняю`, `сохранено`, `ошибка`, время последнего сохранения. Готово в `0.1.0`.
2. Черновики и восстановление после перезагрузки.
3. Шаблон дня для пустой записи. Готово в `0.1.0`.
4. Недельный обзор без AI. Готово в `0.1.0`.
5. Напоминание сделать экспорт.
6. Локальные PWA-уведомления.

Риски:

- автосохранение не должно лагать при вводе;
- подсказки не должны раздражать;
- уведомления зависят от разрешений браузера.

### 4. Приватность

Сначала честная UX-блокировка, потом криптография.

Порядок:

1. Добавить экран “где хранятся данные”.
2. Убрать иллюзию реального логина или объяснить, что вход локальный.
3. Добавить PIN-код как локальную блокировку.
4. Добавить автоблокировку после бездействия.
5. Подготовить Web Crypto шифрование.
6. Добавить зашифрованный экспорт.

Риски:

- потеря пароля при настоящем шифровании означает потерю данных;
- перед миграцией хранилища нужен backup;
- нельзя подключать внешнюю синхронизацию без модели секретов.

### 5. Функциональный рост

Добавлять только то, что усиливает привычку вести дневник.

Приоритет:

1. Настроение дня.
2. Энергия, фокус, стресс, сон.
3. Избранные записи.
4. Группировка истории по месяцам.
5. Полноценный экран календаря.
6. Режим фокуса для письма.
7. Настраиваемые шаблоны.

Риски:

- не перегрузить главный экран;
- новые поля должны иметь хорошие значения по умолчанию;
- статистика должна быть полезной, а не просто красивой.

### 6. Качество проекта

Порядок:

1. Перейти на ES modules.
2. Добавить Vite или оставить лёгкий сервер, если сборщик пока не нужен.
3. Добавить unit-тесты для чистых функций.
4. Добавить smoke/e2e-проверку ключевых сценариев.
5. Проверить PWA через localhost.
6. Подготовить деплой.

Будущие инструменты:

- Vite;
- Vitest;
- fake-indexeddb;
- Playwright.

## Ближайшие 3 задачи

1. Начать модульный разнос с чистых функций.
2. Добавить черновики и восстановление после перезагрузки.
3. Подготовить IndexedDB-слой.
